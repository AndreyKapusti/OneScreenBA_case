## User Story
Я, как менеджер по работе с Wildberries, хочу видеть таблицу с остатками и датами последних продаж в разрезе каждой поставки, чтобы своевременно выявлять зависшие товары и минимизировать потери от замороженного капитала в нереализуемых остатках

## Концепция решения

Дёрнуть три API-ручки (список поставок, состав поставки, продажи), собрать данные в единую таблицу с расчетом остатков и дней без продаж, отфильтровать по порогу N, выявить зависшие остатки.

---

## 1. Цель и назначение

**Цель** — выявить товары, которые поступили на склад Wildberries в рамках конкретных поставок, но длительное время не продаются.

**Назначение** — предоставить пользователю таблицу, в которой по каждой поставке отражены:
- принятое количество товара;
- количество проданных единиц;
- текущий реальный остаток;
- дата последней продажи.

---

## 2. Источники данных

Документация: `https://dev.wildberries.ru/`

- **Список поставок:**  
`/api/v1/supplies`  
Ключевые поля:  
supplyID - ID поставки, factDate - дата передачи товара, statusID == 6 - статус завершённой поставки (товар принят на склад)

- **Состав поставки**  
`/api/v1/supplies/{ID}/goods`  
ID в запросе - supplyID из списка поставок
Ключевые поля:  
nmID - ID товара (уникален для типа товара [в поставкке может быть 10 футболок, у всех одинаковый nmID]), quantity - количество товаров с nmID в поставке  

- **Список продаж:**  
`/api/v1/supplier/sales`  
Ключевые поля:  
nmId, date, saleID - ID продажи / возврата для учёта реального баланса (непроданого) товара, incomeID - ID поставки == supplyID из списка поставок

---

## 3. Алгоритм обработки данных

### 3.1. Сбор данных

1. **Список поставок** — получаем все поставки со статусом `statusID == 6` (завершенные, товар принят на склад). Для каждой поставки фиксируем `supplyID` и `factDate`.

2. **Состав поставки** — по каждому `supplyID` получаем список товаров с полями:
   - `nmID` — идентификатор товара
   - `quantity` — количество единиц данного товара в поставке

3. **Список продаж** — получаем все операции за период с полями:
   - `nmId` — идентификатор товара
   - `date` — дата операции
   - `saleID` — идентификатор операции (для различения продаж и возвратов)
   - `incomeID` — идентификатор поставки, к которой относится операция (связка с `supplyID`)

4. **Доп параметры (вводятся пользователем)**
    - `N` - пороговое количество дней без продаж, после которого считать, что товар потерялся
    - `Фактическая дата поставки` - для определния ID поставки

### 3.2. Связывание данных

Связка выполняется напрямую по полю:
- `incomeID` (из продаж) = `supplyID` (из поставок)

За конкретной поставкой необходимо закрепить nmID, quantity

### 3.3. Расчет итоговых показателей

Для каждой поставки в разрезе `nmId` рассчитываем:

| Показатель | Формула / источник |
|------------|-------------------|
| **Принято (шт)** | `quantity` из состава поставки |
| **Продано (шт)** | Сумма операций с положительным значением (продажи) по `incomeID` = `supplyID` и `nmId` = `nmID` |
| **Возвраты (шт)** | Сумма операций с отрицательным значением (возвраты) по тем же связкам |
| **Текущий остаток (шт)** | `Принято — Продано + Возвраты` |
| **Дата последней продажи** | Максимальная `date` среди продаж для данной поставки и `nmId` |
| **Дней без продаж** | `Текущая дата — Дата последней продажи` (в днях) |

### 3.4. Выявление зависших остатков

Поставка считается проблемной (зависший остаток), если:
- `Текущий остаток > 0`
- `Дней без продаж >= N`, где N — пороговое значение

---

## 4. Требования

### 4.1. Отображение таблицы

Таблица должна содержать следующие колонки:

| Колонка | Описание |
|---------|----------|
| **Дата поставки** | `factDate` |
| **Номер поставки** | `supplyID` |
| **Артикул WB** | `nmID` |
| **Принято, шт** | `quantity` из состава поставки |
| **Продано, шт** | сумма продаж по `incomeID` = `supplyID` |
| **Возвраты, шт** | сумма возвратов по `incomeID` = `supplyID` |
| **Текущий остаток, шт** | Принято — Продано + Возвраты |
| **Дата последней продажи** | максимальная дата продажи по этой поставке |
| **Дней без продаж** | расчетное поле |
| **Статус** | «Норма» / «Зависший остаток» (с цветовой индикацией) |

---

## 5 Критерии приемки

### 5.1. Обязательные (без них решение не принимается)

1. **Корректность связки продаж с поставками**
   - Для выбранной поставки сумма продаж по `incomeID` совпадает с данными в личном кабинете Wildberries (выборочная проверка по 3-5 поставкам).

2. **Корректность расчета остатка**
   - `Остаток = Принято — Продано + Возвраты`. Результат не отрицательный (если отрицательный — ошибка в данных или логике).

---

### 5.2. Желательные

5. **Индикация давности данных**
   - На экране отображается дата последнего обновления данных (например, «Данные на 15.03.2026»).

6. **Устойчивость к ошибкам API**
   - При недоступности API пользователь видит понятное сообщение, а не падение приложения.
